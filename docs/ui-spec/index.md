# UI 设计规范交付文档

> **面向对象**: UI 设计师 → 前端开发  
> **用途**: 规范 UI 设计师交付给前端开发的设计产出物，确保组件库开发有据可依  
> **参考标准**: 飞书开放平台设计规范（[Breadcrumb 示例](https://open.feishu.cn/document/design-specification/component---navigation/breadcrumb)）

---

## 📐 规范体系总览

```
┌─────────────────────────────────────────────────┐
│                  01. Design Tokens               │  ← 原子级：全局设计变量
│         颜色 / 字号 / 间距 / 圆角 / 阴影           │     所有组件的基础
├─────────────────────────────────────────────────┤
│                  02. 组件设计规范                  │  ← 分子级：每个组件的完整规范
│     Breadcrumb / Table / Form / Modal / ...      │     包含结构、样式、状态、交互
├─────────────────────────────────────────────────┤
│                  05. 全局交互规范                  │  ← 全局级：跨组件的通用交互行为
│       动画曲线 / 加载态 / 空状态 / 错误态          │     保证体验一致性
└─────────────────────────────────────────────────┘
```

## 🚀 使用流程

### 对 UI 设计师

```
1. 阅读 01-design-tokens.md → 理解并导出 Figma Variables
2. 阅读 02-component-spec-template.md → 了解每个组件需要交付什么
3. 参照 03 / 04 示例 → 按相同颗粒度填写其他组件
4. 交付前 → 用 06-delivery-checklist.md 自查
```

### 对前端开发

```
1. 收到 01-design-tokens.md → 映射为 CSS Variables，注入全局样式
2. 收到每个组件的规范 → 对照开发组件，确保每个状态都有对应样式
3. 按 05-interaction-spec.md → 封装全局动画/反馈 mixin
```

## 🔗 与飞书设计规范的对应关系

飞书的组件规范（如 Breadcrumb）相当于本体系中的 **02 层（单组件规范）**，但它**缺少**以下内容，这正是我们要求 UI 额外补充的：

| 飞书已有的 | 我们需要额外补充的 |
|-----------|-------------------|
| 使用规则（何时用） | ✅ 已有 |
| 组成要素 | ✅ 已有 |
| 通用样式规则 | ✅ 已有 + **标注对应 Design Token 变量名** |
| 超长/边界处理 | ✅ 已有 |
| — | ❌ **组件状态矩阵（default/hover/active/disabled/error/loading）** |
| — | ❌ **各状态的视觉规格（颜色值必须引用 Token）** |
| — | ❌ **与相邻组件的间距关系** |
| — | ❌ **亮色/暗色双主题对比** |
| — | ❌ **异常场景（空数据、接口报错、权限不足）** |

---

## 📐 设计原则

1. **Token 先行**: 任何硬编码的颜色/字号/间距数字都属于违规，必须引用 Design Token
2. **状态完备**: 每个组件必须覆盖所有交互状态，不允许有"UI 没给就先空着"的情况
3. **主题双轨**: 每个规范必须同时给出 default（亮色）和 dark（暗色）两套值
4. **边界可见**: 空数据、超长文本、极端尺寸等情况必须有明确处理方案
