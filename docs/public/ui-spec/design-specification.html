数据中台平台 — 设计规范 v2.0

本文档定义了「数据中台」平台的设计规范，涵盖设计原则、Design Token 体系、主题系统架构、 组件使用规范、布局标准和扩展指南。适用对象为前端开发人员与 UI/UX 设计师。 规范基于项目当前技术栈（Vue 3 + Ant Design Vue 4 + Less + WindiCSS）制定， 并对现有的设计资产进行了系统化梳理和分层重构。

版本说明：v2.0 对原有的扁平化 CSS 变量命名进行了分层重构， 建立了 Global → Component → Feature 三级 Token 体系。 旧变量名仍可使用，新代码请优先采用新命名。详见变量迁移对照表。

设计原则

1. 清晰优先

数据中台承载大量结构化数据与复杂业务流程，界面设计以确保信息清晰传达为首要目标。 避免过度装饰，减少视觉噪音。

2. 一致性

相同功能的交互模式、视觉样式、文案表述在整个平台中保持一致。 降低用户学习成本，提升操作效率。

3. 主题可切换

平台支持四套主题（信息管理 / 蓝色 / 灰色 / 暗黑），所有组件和页面必须基于 CSS 变量构建， 确保在任意主题下均可正常显示。不允许在组件中硬编码颜色值。

4. 响应式适配

以 1920×1080 为基准设计分辨率，通过 rem 自适应 + 弹性布局 + 最小宽度限制， 覆盖 1366px ~ 2560px 主流屏幕。滚动区域使用自定义滚动条样式。

5. 渐进增强

核心功能不依赖特定浏览器特性，动画/过渡作为体验增强层。

Design Token 层级总览

Token（设计令牌）是设计系统的原子单位。本规范将 Token 分为三个层级：

Global Token（全局）
定义基础设计原子：色彩色板、字体、间距、圆角、阴影。不直接绑定语义。
Component Token（组件级）
引用 Global Token，定义具体组件的视觉表现。如表格表头背景、弹窗边框等。
Feature Token（业务级）
引用 Component Token 或 Global Token，为特定业务场景（数据同步、思维导图等）提供定制变量。
核心规则：组件样式和页面样式只能引用 Token 变量， 禁止在样式中硬编码颜色（如 #2575f5、rgba(0,0,0,0.85)）。 需要新颜色时，先判断是否可归入现有 Token，否则新增语义化 Feature Token。

主题系统架构

工作原理

主题系统采用 CSS Class 切换 + CSS 自定义属性 方案：

  1. 每套主题将变量定义在 .theme-{name} 类下
  2. 主题切换时，Pinia Store 操作 <html> 元素的 class
  3. 所有组件通过 var(--token) 引用变量，浏览器自动重新计算样式
  4. 全局过渡动画（0.3s ease）平滑视觉切换

数据流

用户点击 ThemeSwitch
  → themeStore.setTheme('dark')
    → localStorage.setItem('theme', 'dark')
    → applyTheme('dark')
      → html.classList.remove('theme-default/blue/grey')
      → html.classList.add('theme-dark')
        → CSS 级联生效：所有 var(--*) 切换为 dark 主题值
    → Vue watch 触发（layouts/index.vue）
      → getCssVariable() 读取 7 个 Computed CSS 变量
      → 传给 <a-config-provider :theme="{token:{...}}">
        → Ant Design 组件内部变量同步更新

新建主题指南

步骤

  1. 创建变量文件：在 src/assets/styles/themes/ 下新建 your-theme.less
  2. 定义全部 CSS 变量：参照 default.less 的结构，按 Global → Component → Feature 三层覆盖所有变量。最低要求覆盖全部 Global Token 和 Component Token。
  3. 注册到样式入口：在 src/assets/styles/main.less 中 @import "themes/your-theme";
  4. 注册到 Store：在 src/store/modules/theme.js 的 themes 数组中添加 { name: 'your-theme', label: '你的主题名' }
  5. 更新切换 UI：ThemeSwitch.vue 会自动从 themes 数组渲染，无需额外修改

强制检查清单

新主题必须满足以下条件：
  • ✅ 覆盖 全部 130+ 个 CSS 变量（不可遗漏任何一个）
  • ✅ 亮色主题与暗色主题的对比度分别满足 WCAG AA 标准
  • ✅ 表单、表格、弹窗在所有状态下可清晰辨认（禁用态、错误态、悬停态）
  • ✅ 在 4 个核心页面验证：数据表格页、表单弹窗页、树形列表页、数据同步配置页
  • ✅ 滚动条与背景有足够对比度，用户可以找到并拖拽滚动条

Ant Design 主题联动

项目通过 <a-config-provider :theme="antdTheme"> 将项目自身的 CSS 变量桥接到 Ant Design Vue 4 的 Design Token 系统：

// src/layouts/index.vue
watch(
  () => themeStore.currentTheme,
  () => {
    antdTheme.value['token'] = {
      colorBgBase:     themeStore.getCssVariable('--color-bg-container'),
      colorPrimary:    themeStore.getCssVariable('--color-primary'),
      colorSuccess:    themeStore.getCssVariable('--color-success'),
      colorWarning:    themeStore.getCssVariable('--color-warning'),
      colorError:      themeStore.getCssVariable('--color-error'),
      colorTextBase:   themeStore.getCssVariable('--color-text-primary'),
      borderRadius:    Number(themeStore.getCssVariable('--radius-base')?.replace('px', ''))
    }
  },
  { immediate: true }
)
扩展建议：当前仅同步了 7 个 Token。后续可根据需要增加 colorBgContainer、colorBorder、fontSize 等更多 Token 映射， 使 Ant Design 组件更精准地匹配自定义主题。

使用原则

  • 优先使用共享组件：表格、表单、弹窗等标准场景必须使用 Basic* 组件，不允许直接裸用 ant-design-vue 组件

  • 插槽透传：所有 Basic* 组件自动转发 $slots
  • 通过 defineExpose 暴露命令式 API

  1. 在规范中确定 Token 命名（--comp-{component}-{property}）
  2. 在 4 个主题文件中分别添加该变量（保持相同的变量名，不同的值）

  4. 若需联动 Ant Design 内部 Token，在 layouts/index.vue 的 watch 中添加映射

扩展指南：组件

新增共享组件

  1. 在 src/components/ 下创建目录
  2. 组件必须导出清晰的 Props / Events / Slots / Expose
  3. 若为 Ant Design 组件的二次封装，遵循 Basic* 命名约定：Basic{Name}.vue
  4. 通过 v-bind="$attrs" 透传未声明 Props
  5. 通过 v-for="$slots" 透传插槽
  6. 通过 defineExpose 暴露命令式方法

  2. 在 defaultComponentProps 中定义默认属性
  3. 确保组件支持 v-model:value 或 v-model:checked

👤 前端开发工程师

你是本规范的主要使用者。日常开发中需要频繁参考 Token 定义和组件规范。

日常开发流程

  1. 写样式前：先查 色彩 Token → 组件 Token → 业务 Token，确认所需变量已存在
  2. 写组件时：优先使用 BasicTable / BasicForm / BasicModal，参考 组件规范总览
  3. 写表单时：查 表单规范 确认 22 种组件类型和 Schema 写法
  4. 颜色硬编码零容忍：所有颜色必须用 var(--xxx)，不确定用哪个 Token 时查本文档
  5. 提交前：在 4 套主题下快速过一遍页面，确保无颜色异常

前后端协同维护规范

设计规范的有效性取决于持续的维护和更新。以下是前端开发与 UI 设计师协同维护规范的机制和流程。

协同原则

  1. 规范是唯一真相源：设计稿和代码必须以本文档定义的 Token 和组件为准，不一致时应更新本文档
  2. 双向驱动：设计师提出的视觉变更 → 影响 Token 定义；开发提出的组件封装 → 影响组件规范
  3. 变化可追溯：文档放在 Git 仓库中，所有变更通过 PR 评审，记录修改原因和影响范围

场景一：设计师新增/修改视觉风格

触发条件：UI 设计师在 Figma/MasterGo 中调整了某个颜色、字体大小或间距， 需要同步到代码和规范文档。

流程

  1. 设计师在 Figma/MasterGo 中标注变更涉及的设计变量名（Token 名称），导出变更清单
  2. 设计师在本文档中找到对应 Token（如 --color-primary），在 GitLab/PR 中提交文档更新
  3. 前端开发根据文档变更，在 4 个主题 Less 文件中修改对应 CSS 变量值
  4. 前端开发在 4 套主题下截图验证，贴到 PR 评论中
  5. 设计师Review 截图，确认视觉还原度后 Approve PR
关键约定：设计师不能直接给开发一个 "这个按钮换个蓝色" 的口头描述。 必须给出 Token 名称和新色值，例如：「--color-primary 从 #0D6EF0 改为 #1677FF」。 这样做的好处是：一次修改，四套主题、所有引用该 Token 的组件自动生效。

场景二：开发新增组件或页面

触发条件：前端开发新增了一个功能模块或共享组件， 其中涉及新的视觉样式（颜色、间距、状态等）。

流程

  1. 前端开发先判断新样式能否用现有 Token 覆盖：
    • 能用 → 直接用，无需更新文档
    • 不能用 → 评估是新增 Feature Token 还是 Component Token
  2. 前端开发在本文档中新增 Token 定义（按命名规范），在 4 个主题文件中添加对应变量
  3. 前端开发在 PR 描述中 @设计师，附上 Token 新增说明和页面截图
  4. 设计师确认新 Token 的命名和默认值符合设计意图，Review 通过

场景三：规范文档本身的迭代

触发条件：规范需要调整（如新增章节、修改 Token 层级、调整组件 API 等）。

流程

  1. 在 GitLab 上创建 Issue，描述变更原因和影响范围
  2. 负责人（前端 Leader 或设计 Lead）Assign 给对应人员
  3. 修改文档 → 提交 PR → 前端 Leader + 设计 Lead 双人 Review
  4. 合并后通知全员（群消息 / 周会同步）

修改一个 Token 前，请用以下方法自查影响范围：
